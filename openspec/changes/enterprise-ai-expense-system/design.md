## Context / 上下文

该项目面向“前端工程师转 AI 工程师”的实战学习场景，目标不是只完成一个 Demo，而是完整经历 AI 应用从产品拆解、数据建模、后端服务、前端工作台、Agent 编排到上线验证的过程。系统需要同时支持票据上传与识别、报销单生成、审批流、组织权限和知识增强解释，因此属于典型的跨前后端与 AI 编排的复合型工程。

当前仓库尚未实现该系统，需要先建立一套可以分阶段推进的架构：前端使用 React + Ant Design 构建业务工作台，后端使用 FastAPI + SQLModel 提供 API 和数据模型，AI 流程使用 LangGraph 组织 OCR、字段抽取、规则校验和解释节点，知识增强使用向量数据库承载报销制度与常见问题。

## Goals / Non-Goals / 目标与非目标

**Goals / 目标：**
- 建立企业级 AI 报销系统的端到端技术方案，覆盖前端、后端、数据库、向量检索和 Agent 编排。
- 采用适合学习与扩展的模块化架构，让每个阶段都能独立交付并验证。
- 用结构化数据模型承载票据、报销单、审批、组织和知识库实体。
- 将 AI 能力设计为可替换组件，支持后续替换 OCR、LLM、Embedding 与向量数据库实现。
- 把项目拆解为适合前端工程师学习的实现顺序，先交付业务骨架，再叠加 AI 能力。

**Non-Goals / 非目标：**
- 本次设计不追求一次性覆盖复杂的企业财务规则，如多币种报税、跨法人结算和 ERP 深度集成。
- 本次设计不要求实现生产级高可用部署方案，如多区域容灾或复杂消息队列体系。
- 本次设计不绑定单一模型厂商，避免把架构与某个闭源服务强耦合。

## Decisions / 关键设计决策

### 1. 前端采用 Vue 3 + Ant Design Vue

当前实现选择 Vue 3 + Ant Design Vue 作为前端技术栈。这样既能满足企业后台场景对表单、表格和工作台布局的需求，也与当前用户明确提出的实现偏好保持一致，便于在熟悉的前端语境下把更多精力投入到后端和 AI 工程能力建设上。

备选方案：
- React + Ant Design：同样可行，后续如果有团队统一技术栈要求，也可以平滑切换。

### 2. 后端采用 FastAPI + SQLModel

FastAPI 适合快速构建类型清晰的 API，并天然适合 Python AI 生态。SQLModel 可以让 ORM 模型与接口 Schema 更统一，降低学习门槛，便于前端背景的开发者理解后端数据边界。

备选方案：
- Django / Django REST Framework：功能更全，但学习与约束更重。
- Node.js + NestJS：更贴近前端背景，但会削弱进入 Python AI 生态的价值。

### 3. 业务数据与检索数据分层存储

关系型数据库负责用户、组织、票据、报销单、审批流和审计日志；向量数据库负责报销制度、票据示例、审批规则解释材料的向量化存储。这样可以避免把事务型数据与检索型数据混在一起，保持数据职责清晰。

备选方案：
- 仅用 PostgreSQL + pgvector：可降低组件数量，适合 MVP。
- 单独引入 Milvus：更接近企业向量检索架构，利于后续扩展。

### 4. AI 流程使用 LangGraph 状态图编排

票据识别并不是单次模型调用，而是“上传 -> OCR -> 字段抽取 -> 规则校验 -> 异常解释 -> 结果汇总”的状态流。LangGraph 能更清晰地表达节点、状态和回退路径，也更适合教学，让学习者理解 Agent 并非魔法，而是可编排的工作流。

备选方案：
- 仅用 LangChain 顺序链：实现更快，但不利于表达复杂状态流与人工复核节点。

### 5. 报销规则解释采用 RAG 而不是硬编码 Prompt

当系统需要解释“为什么这张票据异常”或“该费用能否报销”时，仅靠固定 Prompt 难以维护。将制度文档切片入库并检索增强，可以提高可解释性，也让项目具备知识库工程能力。

备选方案：
- 规则全部硬编码在服务端：实现简单，但扩展性与解释性较差。

### 6. 项目采用分阶段交付

第一阶段先交付账户体系、组织权限、票据上传和报销单基础 CRUD；第二阶段接入 OCR 与字段抽取；第三阶段引入 LangGraph 编排与知识增强；第四阶段再补充审计、评估与部署。这样既适合学习，也更有利于持续验证。

## Risks / Trade-offs / 风险与权衡

- [AI 依赖较多，初学者容易被环境问题卡住] → 先实现可运行的业务骨架，再逐步接入 OCR、LLM 与向量库。
- [LangGraph 增加理解成本] → 先用单节点或双节点流程起步，再扩展到多智能体图。
- [向量数据库带来部署复杂度] → MVP 可先使用 PostgreSQL + pgvector，后续再切换到 Milvus。
- [票据识别准确率不稳定] → 保留人工编辑与复核界面，避免把模型输出直接当成最终结果。
- [权限模型过早复杂化] → 第一版只实现组织、角色、用户三级控制，细粒度资源权限后置。

## Migration Plan / 分阶段落地计划

1. 创建后端服务骨架、数据库模型与基础认证能力。
2. 创建前端管理台骨架，完成登录、上传、报销单列表与详情页面。
3. 接入文件存储与 OCR，打通票据上传到结构化字段抽取链路。
4. 引入 LangGraph，拆分识别、校验、解释与汇总节点。
5. 引入知识库与向量检索，为规则解释与问答提供支持。
6. 增加评估、日志、审计与部署配置。

回滚策略：
- AI 流程节点通过配置开关控制，异常时可降级为“仅上传 + 人工填写”模式。
- 知识检索异常时，保留基础规则校验接口，避免主流程不可用。

## Open Questions / 待确认问题

- 审批流程第一版是否只支持单级审批，还是预留多级审批配置。
