# 后端骨架文件说明（前端视角版）

这份文档是给你这种有多年前端经验、刚开始系统学习后端和 AI 工程的人准备的。

目标不是让你死记硬背 FastAPI 的语法，而是帮你把它和熟悉的前端概念一一对应起来，这样你会更容易理解这一步到底搭了什么。

---

## 一、第一步我们到底完成了什么

我们完成的不是“业务功能”，而是 **后端项目地基**。

如果拿前端来类比，这一步相当于：

- 用 `Vite` 或 `Next.js` 初始化了一个 React 项目
- 配好了项目目录结构
- 配好了环境变量
- 配好了基础路由
- 写了一个 `/health` 这种最简单的接口页面
- 再补了一条基础测试，确认项目真能跑

也就是说，这一步解决的是：

- 项目怎么启动
- 配置从哪里读
- 路由放哪里
- 数据库以后接哪里
- 测试怎么写

这就是后面所有业务开发的起点。

## 三、每个文件是干什么的

## 1. `pyproject.toml`

这是 Python 项目的核心配置文件。

你可以把它理解成后端世界里的：

- `package.json`
- 再加上一部分 `vite.config.ts`
- 再加上一部分测试配置

它现在主要做了几件事：

- 定义项目名和版本
- 声明 Python 版本要求
- 声明运行依赖
- 声明开发依赖
- 配置测试目录

当前核心依赖：

- `fastapi`：后端框架
- `sqlmodel`：ORM + 数据模型
- `pydantic-settings`：环境变量配置管理
- `uvicorn`：启动 FastAPI 的服务进程
- `pytest`：测试框架

前端类比：

- `dependencies` 很像前端里的运行时依赖，比如 `react`、`axios`
- `optional-dependencies.dev` 很像前端里的 `devDependencies`

---

## 2. `.env.example`

这是环境变量模板。

作用是告诉你这个项目运行时需要哪些配置，但不直接放真实敏感值。

比如现在有：

- 应用名
- 运行环境
- 端口
- API 前缀
- 数据库地址

前端类比：

- 很像前端项目里的 `.env.development`
- 或者 `VITE_API_BASE_URL`

区别是：

- 前端环境变量大多最终会影响浏览器端代码
- 后端环境变量更多控制服务本身，比如数据库连接、模型 key、端口、日志级别

---

## 3. `.gitignore`

这个文件告诉 Git 哪些文件不要提交。

目前忽略了：

- `.venv/`：Python 虚拟环境
- `__pycache__/`：Python 缓存文件
- `.pytest_cache/`：测试缓存
- `.env`：本地敏感配置

前端类比：

- 和你项目里的 `node_modules/`、`dist/`、`.env.local` 不提交是一个逻辑

---

## 4. `README.md`

项目说明文档。

它不是给程序运行的，而是给开发者看的。

目前写了：

- 这个项目是什么
- 当前骨架包含什么
- 推荐先看哪些文件
- 本地怎么启动

前端类比：

- 和前端项目根目录的 `README.md` 完全同一职责

---

## 5. `app/`

这是后端应用源码目录。

你可以把它理解成前端里的 `src/`。

以后主要业务代码都会往这里放：

- 路由
- 配置
- 数据模型
- 服务层
- Agent 工作流
- 数据库访问逻辑

---

## 6. `app/__main__.py`

这是命令行启动入口。

你执行：

```powershell
python -m app
```

实际上就是从这里进去。

它现在做的事情很简单：

- 读取配置
- 用 `uvicorn` 启动 FastAPI

前端类比：

- 很像前端项目里的 `npm run dev`
- 或者 dev server 的启动入口脚本

你平时不会经常改它，但它决定“项目怎么跑起来”。

---

## 7. `app/main.py`

这是 FastAPI 应用真正被创建的地方，是后端骨架里最关键的文件之一。

它做了几件事：

- 创建 `FastAPI` 应用实例
- 注册统一路由
- 定义根路径 `/`
- 预留生命周期钩子 `lifespan`

前端类比：

- 很像 React 里的 `main.tsx`
- 或者应用级 `App.tsx` + 根级 provider 注册位置

可以这样理解：

- `FastAPI(...)` 类似前端里创建应用根实例
- `include_router(...)` 类似把多个模块路由统一挂到主应用上
- `lifespan` 类似应用启动时初始化全局资源

比如以后这些东西常常会在这里挂进去：

- 数据库初始化
- Redis 初始化
- LLM 客户端初始化
- 向量数据库连接

---

## 8. `app/api/router.py`

这个文件负责聚合所有 API 路由。

现在它把 `health.py` 注册了进来，后面还会继续注册：

- 认证路由
- 用户路由
- 报销单路由
- 文件上传路由
- 知识库路由

前端类比：

- 很像前端的 `router/index.ts`
- 或者把多个页面路由模块统一汇总的地方

区别在于：

- 前端路由通常是“页面地址 -> 页面组件”
- 后端路由通常是“接口地址 -> 处理函数”

---

## 9. `app/api/routes/health.py`

这是一个最简单的路由文件，目前只有一个接口：

- `GET /api/v1/health`

这个接口的意义不是业务，而是验证：

- 服务是否启动成功
- 路由是否注册成功
- 配置是否读取成功

前端类比：

- 很像你前端项目初始化时，先做一个最简单的 demo 页面
- 或者先打一个 `/ping` 页面验证路由系统正常

在真实项目里，`health` 常常给：

- 运维探活
- 容器健康检查
- 监控系统判断服务在线状态

---

## 10. `app/core/config.py`

这个文件负责读取并管理配置。

这里定义了 `Settings`，把环境变量映射成 Python 对象。

比如：

- `APP_HOST`
- `APP_PORT`
- `APP_DATABASE_URL`

都会被读取后变成代码里可直接访问的字段。

前端类比：

- 很像你在前端封装一个 `config.ts`
- 在里面统一读取 `import.meta.env`

区别是：

- 前端常用 `import.meta.env.VITE_XXX`
- 这里是 `BaseSettings` 自动读取系统环境变量或 `.env`

为什么要这么做？

因为配置不应该写死在代码里。

后面切换环境时，比如：

- 本地环境
- 测试环境
- 生产环境

只改配置，不改代码。

这就是工程化的基本功。

---

## 11. `app/db/session.py`

这个文件现在先放了数据库引擎创建逻辑。

目前还没进入真正的业务建模阶段，但它先把数据库入口占住了。

前端类比：

- 有点像你先封装了一个 `request.ts` 或 `axios.ts`
- 虽然业务接口还没写完，但统一入口先准备好了

后面这个目录还会继续长出：

- 数据模型
- Session 管理
- 数据库初始化函数
- 迁移相关逻辑

你可以先把 `engine` 理解成“数据库连接总入口”。

---

## 12. `tests/test_health.py`

这是第一条自动化测试。

它做的事情是：

- 用测试客户端启动应用
- 请求 `/api/v1/health`
- 断言返回状态码是 `200`
- 断言返回数据里 `status == "ok"`

前端类比：

- 很像你用 `Vitest` 或 `Jest` 写一个最基础的接口/组件测试
- 或者对一个页面渲染结果做断言

它的意义不是测试业务复杂度，而是验证：

- 项目测试链路通了
- 路由通了
- 应用实例创建没问题

这在真实工程里非常重要。

---

## 13. `__init__.py` 文件是干嘛的

你会看到很多目录下都有 `__init__.py`。

它的主要作用是告诉 Python：

“这个目录是一个可导入的包。”

前端类比：

- 有点像目录里的 `index.ts`
- 也有点像在告诉构建工具“这是一个模块边界”

虽然很多时候它内容很少，但它对 Python 包结构是基础设施。

## 四、用前端思维重新理解这套后端结构

你可以先把它映射成下面这样：

| 后端文件/概念 | 前端近似概念 | 作用 |
| --- | --- | --- |
| `pyproject.toml` | `package.json` | 管理依赖、项目元信息、测试配置 |
| `.env.example` | `.env.example` / `VITE_XXX` 模板 | 管理环境变量 |
| `app/main.py` | `main.tsx` / `App.tsx` | 创建应用根实例 |
| `app/api/router.py` | `router/index.ts` | 聚合路由 |
| `app/api/routes/*.py` | 页面路由模块 / API 模块 | 定义具体接口 |
| `app/core/config.py` | `config.ts` | 统一读取配置 |
| `app/db/session.py` | `request.ts` / `apiClient.ts` 的“基础设施层” | 统一数据库入口 |
| `tests/test_health.py` | `*.test.ts` | 自动化测试 |

这能帮你建立一个关键认知：

**后端并不神秘，它也是一个“有入口、有配置、有路由、有基础设施”的应用。**

只是：

- 前端主要面向浏览器和用户界面
- 后端主要面向数据、接口、权限和服务运行

## 五、为什么第一步不直接开始写业务

这是很多前端转后端时最容易忽略的点。

你可能会想：

- 为什么不直接写登录？
- 为什么不直接写报销单接口？
- 为什么要先写 health？

答案是：**先搭地基，后面才不会反复返工。**

如果没有这套骨架，后面你会遇到这些问题：

- 配置散落在各处
- 路由组织混乱
- 数据库入口不统一
- 测试没法写
- 启动方式不稳定

工程项目一旦变大，这些问题会迅速拖慢你。

所以第一步本质是在做：

**项目工程化初始化。**

## 六、从 AI 工程师视角看，这一步为什么重要

因为 AI 工程师不是“只会调模型的人”。

真正的 AI 工程开发一定要落到：

- 服务怎么启动
- 模型怎么配置
- API 怎么暴露
- 数据怎么存
- 工作流怎么组织
- 日志和测试怎么做

你现在做的这一步，其实是在补你从前端工程师走向 AI 应用工程师最重要的一块短板：

**后端服务工程化能力。**

后面无论你接：

- OCR
- LLM
- Embedding
- LangGraph
- 向量数据库

最终都要挂到这套骨架里。

## 七、你现在最应该重点理解哪几个文件

如果你只先看 4 个文件，优先顺序建议是：

1. `app/main.py`
2. `app/core/config.py`
3. `app/api/router.py`
4. `app/api/routes/health.py`

理解了这四个，你就已经掌握了：

- 应用怎么创建
- 配置怎么进入代码
- 路由怎么组织
- 一个接口怎么写出来

这四个点打通后，后面加登录、用户、报销单、Agent 工作流都会顺很多。

## 八、下一步你会学到什么

如果继续往下做，下一步通常有两个方向：

### 方向一：继续做前端骨架

你会把熟悉的 React 管理台搭起来，然后和后端联调。

这条路线的优点是熟悉感强。

### 方向二：继续做后端数据模型与认证

你会学到：

- 用户、组织、角色怎么建模
- FastAPI 里依赖注入怎么用
- 登录认证接口怎么写
- 权限怎么做

这条路线更贴近“后端/AI 工程能力提升”。
